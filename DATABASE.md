# Database Schema & Structure

The core of the GoRide Search Engine is powered by PostgreSQL augmented with PostGIS and pg_trgm.

## Schema Overview: `goride.places`

The database consists of a primary table `places` generated via SQLAlchemy models.

### Important Columns

- `id` (Integer, Primary Key)
- `place_id` (String, Unique): A deterministic ID used to prevent duplicates during CSV imports (usually a hash of name and coordinates).
- `name` (String): The primary name of the location.
- `display_name` (String): A formatted full address (e.g., "Apollo Hospital, Greams Road, Chennai").
- `latitude` / `longitude` (Float)
- `category` (String): Used for intent routing (e.g., "hospital", "atm", "bus_stop").
- `district`, `state`, `country` (String): Administrative bounds.
- `pincode` (String): 6-digit Indian postal code.
- `search_count` (Integer): Incremented whenever a user explicitly selects a result. Used to boost popular locations.
- `geom` (Geometry): A PostGIS `POINT(lon lat)` column used for rapid spatial distance calculations.

### Indexes

To guarantee sub-millisecond query performance, the following indexes exist:
- **Spatial Index:** GIST index on `geom` for bounding box and nearest-neighbor queries (`<->` operator).
- **Trigram Index:** GIN index using `pg_trgm` on `name` and `display_name`. Allows highly optimized fuzzy matching (`ILIKE '%term%'`).
- **B-Tree Indexes:** On `category` and `pincode` for fast exact matches.

## Migrations

Database schema changes are managed by **Alembic**.
- Migration scripts reside in `backend/alembic/versions/`.
- To apply migrations: `alembic upgrade head`.

## Data Ingestion & CSV Import

Data is ingested into `places` from curated CSV files:
- `hubs_v5.csv`: The primary dataset of GoRide operational hubs and locations.
- `manual_places.csv`: A secondary dataset generated by user feedback via the "Add Missing Place" frontend feature.

### Missing Place Feedback Loop (manual_places.csv)

When a user searches for an address and cannot find it, they use the interactive "Add Missing Place" UI.
1. The backend geocodes the address using Nominatim (`GeocoderService`).
2. The user adjusts the pin on the frontend.
3. The frontend submits the final coordinates to `POST /places/manual`.
4. The `MissingPlaceService` simultaneously:
   - Inserts the record directly into the PostgreSQL `places` table for immediate searchability.
   - Appends the raw record via `CSVAppendService` to `backend/data/manual_places.csv` ensuring it is permanently captured for future DB rebuilds.
